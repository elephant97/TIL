### 웹이란?
> 웹은 전 세계적으로 연결된 컴퓨터 네트워크를 통해 정보를 공유하는 거대한 시스템,     
* 웹은 인터넷을 기반으로 하며 다양한 종류의 정보와 서비스를 제공하는 웹 사이트들로 구성되어 있다.
* 정보는 웹 페이지 라는 형태로 제공되며 HTML로 작성되고, 사용자들은 웹 브라우저를 통해 이러한 웹 페이지에 접근할 수 있다.

### request 변수는 무엇이며, 왜 사용되는가?
> 파라미터접근 헤더 세션관리 클라정보 서블릿경로, 응답MIME/JSON 컨텐츠타입설정, 응답헤더제어, 응답데이터, 쿠키 생성및전송
* Request 변수는 무엇인가?  
  * request 변수는 웹 애플리케이션 개발에 사용되며, 클라이언트로부터 서버로 전송된 HTTP 요청을 나타냄
  * Java servlet API의 HttpServletRequset인터페이스의 인스턴스이며, 웹 서버가 클라이언트 HTTP 요청을 처리할 때 사용됨
* Request변수의 사용 목적
  * 클라이언트 정보 접근
    * 클라이언트가 보낸 데이터에 접근할 수 있다
    * 클라이언트의 정보: 폼데이터, 쿼리파라미터, 헤더 정보, 쿠키 등
  * HTTP 요청 메소드 확인
    * 클라이언트가 보낸 요청이 GET, POST, PUT등 어떤 HTTP 메서드를 사용하는지 확인 가능
  * 세션 관리
    * HTTP는 상태를 유지하지 않는 프로토콜이기 때문에 requset객체를 사용하여 사용자의 세션 정보에 접근하고 관리할 수 있음
  * 사용자 인증 정보 접근
    * 클라이언트의 인증정보에 접근할 수 있음 (예: 로그인 사용자 정보)
  * 서버와 클라이언트 간의 상호작용
    * 서버가 요청을 처리하고 결과를 클라이언트에게 다시 보내는데 필요한 정보를 request객체를 통해 얻을 수 있다
  * 언어 및 콘텐츠 협상
    * 클라이언트가 요청한 언어나 컨텐츠 유형을 파악하여 적절한 응답을 준비하는데 사용 됨

### HTTP Request Header의 캐시컨트롤은 어떤역할을 하며, 왜 필요한가?
> 클라이언트와 서버 사이의 캐싱 메커니즘을 제어하는데 사용 된다. 이 헤더는 요청이나 응답 메세지에 포함되어 캐싱방식을 지정하며,     
> 네트워크 트래픽을 줄이고, 컨텐츠 로딩 시간을 단축시키며 서버 부하를 경감하는데 도움을 줌
* **주요 역할**
  * 캐싱 정책 지정
    * 자신의 캐싱 선호도를 명시할 수 있다.
    * no-cache : 매번 서버에게 컨텐츠의 유효성을 확인하도록 요청 함
    * no-store : 어떠한 캐싱도 하지 않도록 지시함
  * 효율성 향상
    * 캐싱을 적절히 사용함으로써, 자주 접근하는 리소스에 대한 네트워크 요청을 줄일 수 있다.
    * 웹 페이지의 로딩을 단축시키고 사용자 경험을 향상 시킴
  * 대역폭 절약(네트워크 연결을 통해 특정 시간동안 전송될 수 있는 데이터의 최대 양)
    * 캐시 컨트롤을 통해 불필요한 데이터 전송을 줄임으로써 네트워크 대역폭을 절약할 수 있음
  * 서버 부하 감소
    * 캐싱을 통해 서버의 반복적인 요청 처리 부담을 줄일 수 있다.
  * 신선도 및 일관성 유지
    * 캐시된 컨텐츠의 신선도를 유지하는데 사용됨
    * max-age지시어를 통해 컨텐츠가 얼마 동안 캐싱 될 수 있는지 지정할 수 있으며 이를 통해 일관성을 유지함
* **필요 이유**
  * 웹은 다양한 컨텐츠와 대규모 사용자를 수용하기 때문에 네트워크 효율성과 서버 성능을 최적화 하는 것이 필요하며 캐시컨트롤은 이를 돕는 역할을 함
  * 캐시 컨트롤을 통해 데이터 사용량을 줄이고, 사용자 경험을 향상시키며, 웹 애플리케이션의 전빈적인 성능을 관리할 수 있음
    
### <%-- 주석과 <!-- 주석은 두종류가 있는데 차이는 무엇인가?
* `<%--`
  * 해당 주석은 JSP에서 사용되며 JSP페이지가 컴파일되는 동안 서버측에서 처리되는 주석이다
  * 즉 사용자가 페이지 소스보기 버튼을 눌러도 서버측에서 처리되어 클라이언트로 전송되지 않기 때문에 볼 수 없음
  * HTML 소스코드에 나타나지 않는다는 뜻   
* `<!--`
  * HTML문서 내에서 사용되는 주석이며 클라이언트 측에서 처리된다
  * 사용자가 웹 페이지 소스보기 버튼 클릭시 확인 가능
  * 그러나, 브라우저에 의해 렌더링 되지 않기 때문에 사용자에게 직접적으로 보이지는 않는다

### SPA란 무엇인가?
* SPA는 "Single Page Application"의 약자로, 웹 어플리케이션 또는 웹사이트의 한 형태를 지칭함
* SPA는 사용자의 상호작용 동안 전체 페이지를 새로고침하지 않고, 필요한 부분만 동적으로 업데이트 하는 것이 특징
* 사용자가 웹 어플리케이션과 상호작용할 때, 필요한 데이터만 서버로부터 가져와서 현재 페이지에 즉시 반영함
  * **주요 특징**
    * 동적 상호작용
      * JavaScript를 활용하여 사용자의 요청에 따라 페이지의 특정 부분을 업데이트 함
      * 페이지를 전체 재 로드 할 필요가 없어 사용자 경험이 향상 됨
    * 서버 부하 감소
      * 전체페이지가 아닌 필요 데이터만 서버로 부터 불러오기 때문에 서버 부하가 감소함
    * 향상된 사용자 경험
      * 페이지를 새로고침 하지 않기 때문에 빠르고 부드러운 사용자 경험을 제공함
    * 프론트엔드 중심의 개발
      * SPA는 프론트엔드(주로 JavaScript, HTML, CSS)에 중점을 둠
      * React, Angular, Vue.js와 같은 현대적인 JavaScript 프레임워크 라이브러리를 사용하여 개발 됨
  * 단점
    * SEO(검색 엔진) 최적화 문제
      * 동적으로 컨텐츠를 로드하기 때문에 검색 엔진 최적화에 어려움이 있을 수 있음
      * 검색 엔진이 동적으로 로드된 컨텐츠를 효과적으로 인덱싱하지 못할 수 있음
    * 초기 로딩 시간
      * 초기에 애플리케이션의 전체 또는 대부분을 로딩해야하므로 시작시 로딩 시간이 길어질 수 있음
    * 브라우저 호환성 문제
      * 구형 브라우저에서는 JavaScript를 효율적으로 처리하지 못하는 경우가 있어 호환성 문제가 발생할 수 있음
    * 보안 문제
      * 프론트엔드에 많은 로직을 포함하기 때문에, 잘못 설계된 경우 보안 취약점이 발생할 수 있음

### React가 뭘까 왜이렇게 많이 쓰이는걸까
* React
  * Facebook이 개발한 오픈소스 Javascript라이브러리로, 주로 웹 애플리케이션 사용자 인터페이스를 구축하는데 사용됨
  * 주요 목적은 대규모, 동적인 웹 애플리케이션에서 빠르고 효율적인 뷰 레이어를 제공하는 것
* **주요 특징과 장점**
  * 컴포넌트 기반 구조
    * 컴포넌트 기반으로 되어있어, 재사용 가능한 UI부품을 만들 수 있음
    * 개발을 더욱 모듈화하고 관리하기 쉽게 만듬
  * 가상 DOM
    * 실제 DOM대신 가상 DOM을 사용하여 컴포넌트 상태가 변경될 때마다 전체 UI를 가상DOM에 먼저 렌더링한다
    * 이전 가상 DOM과 실제 DOM에 필요한 최소한의 변경만을 적용하여 성능을 향상시킴
  * 선언적 뷰
    * 애플리케이션이 상태에 따라 UI를 선언적으로 설계할 수 있음
    * 상태가 변경되면 React가 자동으로 적절한 컴포넌트를 업데이트함
  * 강력한 커뮤니티 및 생태계
    * React는 널리 사용되며, 크고 활발한 커뮤니티와 강력한 생태계를 가지고 있음
    * 다양한 서드파티 라이브러리, 도구, 확장 기능이 지원됨
  * 다용도성 및 유연성
    * React는 웹 개발뿐만 아니라 모바일 애플리케이션 개발에도 사용될 수 있음
    * 다른 프레임워크나 라이브러리와 쉽게 통합될 수 있음
* **널리 사용되는 이유**
  * 개발의 용이성
    * React의 컴포넌트 기반 접근 방식은 개발자가 복잡한 UI를 쉽게 구성하고 관리할 수 있도록 도와줌
  * 성능
    * 가상DOM이 애플리케이션의 성능을 크게 향상시키며, 동적인 컨텐츠가 많은 대규모 애플리케이션에서 확연하게 드러남
  * 유연성 및 확장성
    * 다른 도구 및 기술과의 통합이 용이함
  * Facebook의 지원
    * Facebook이 React의 개발과 유지보수를 지원하며, 실제로 Facebook자체 제품에도 널리 사용됨
* *현대 웹 개발 특히, 단일 페이지 애플리케이션 개발에 있어 매우 인기가 있다.*

### 리다이렉트는 어떤 과정을 거치나?
* 리다이렉트
  * 웹 브라우저나 다른 클라이언트가 특정 URL로 요청을 보낼 때, 그 요청을 다른 URL로 재 지시하는 방법
  * 웹 사이트의 구조 변경, URL의 단축, 서버 부하 분산, 사용자를 새로운 페이지로 이동시키는 등의 목적으로 활용 됨
* 리다이렉트 과정
  * 초기 요청
    * 사용자 또는 클라이언트가 웹 브라우저를 통해 특정 URL로 요청을 보냄
  * 서버 응답
    * 서버는 해당 요청에 대해 리다이렉트 응답을 보내고, 새로운 URL을 제공함
    * 이 응답에는 특정 HTTP 상태 코드가 포함 됨
  * 브라우저가 새 URL로 요청
    * 브라우저는 서버로부터 받은 URL로 자동으로 요청을 보냄
  * 최종 페이지 로드
    * 새 URL의 서버가 요청을 처리하고 최종 콘텐츠를 사용자에게 보냄
* 리다이렉트 종류
  * 301 리다이렉트 (Moved Permanently)
    * 영구적인 리다이렉트로, 원래의 페이지가 새 URL로 영구적으로 이동했음을 나타냄
    * 검색 엔진 최적화(SEO)에 영향을 주며, 이전 URL의 링크 가치를 새 URL로 전달함
  * 302 리다이렉트 (Found)
    * 일시적인 리다이렉트로, 원래 페이지가 임시로 다른 위치로 이동했음을 나타냄
    * SEO관점에서는 301리다이렉트 만틈 링크 가치를 전달하지 않음
  * 303 리다이렉트 (See Other)
    * 주로 Post요청 후에 사용되며, 클라이언트에게 GET방식으로 다른 URL을 요청하도록 지시함
  * 307 리다이렉트 (Temporary Redirect)
    * 302와 유사하지만, 307은 요청방식을 변경하지 않고 유지함
  * 308 리다이렉트 (Permanent Redirect)
    * 301과 유사하게 영구적인 리다이렉트를 나타내지만, 요청 방식을 변경하지 않고 유지함


### 쿠키란 무엇이며 왜 사용하는가?
* 쿠키
  * 웹사이트가 사용자의 브라우저에 저장하는 작은 데이터 조각
  * 서버가 사용자의 브라우저를 식별하고 사용자의 세션을 유지하는데 사용됨
* 주요 기능
  * 세션관리
    * 사용자가 로그인, 쇼핑 카트 등 사용자의 세션 정보를 저장하고 관리하는데 쿠키가 사용됨
  * 개인화
    * 사용자의 선호도, 테마 설정, 언어 선택 등 개인 맞춤 설정을 기억하여 사용자 경험을 개인화하는데 사용됨
  * 추적
    * 사용자의 웹 사이트 방문 및 상호작용 기록을 추적하여 타겟 광고, 웹사이트 분석 등에 활용됨
* 쿠키 작동 방식
  1. 사용자가 웹사이트 방문 시 웹사이트 서버가 HTTP 응답에 쿠키를 포함시켜 사용자의 브라우저로 보냄
  2. 브라우저가 이 쿠키를 저장하고, 이후 같은 서버로 요청을 보낼 때 마다 쿠키를 HTTP 요청과 함께 서버에 전송 함
  3. 서버가 이 쿠키를 읽어 사용자를 식별하고, 필요한 작업 수행
* 쿠키의 중요성
  * 쿠키를 통해 웹 사이트는 사용자의 방문을 기억하고, 각 사용자에 대한 개별적인 서비스를 제공할 수 있음
  * 세션 관리, 사용자 인증, 사용자 선호도의 저장 등이 웹사이트의 기능성과 사용자 경험 향상
  * 마케팅과 광고 목적으로 사용자의 행동을 분석하는 데에도 쿠키가 중요함
* 문제점
  * 사용자의 프라이버시와 보안에 영향을 줄 수 있음
    
### 쿠키는 왜 도메인 정보가있을까?
* 보안가 데이터 격리, 그리고 쿠키의 올바른 사용을 보장하기 위해 포함되어있다.
* 도메인 정보는 쿠키가 어느 웹사이트에 속하는지를 나타내며 이는 쿠키의 핵심적인 부분이다.
  * 보안
    * 쿠키가 사용될 수 있는 범위 제한
    * 다른 웹사이트가 해당 쿠키에 접근하거나 수정하는 것을 방지하여 사용자의 데이터를 보호함
  * 데이터 격리
    * 서로 다른 웹사이트 간의 데이터 격리를 보장하기 위해 각 쿠키는 도메인별로 저장됨
    * 한 웹사이트의 쿠키가 다른 웹사이트와 혼용되거나 오용되는 것을 방지함
  * 서브 도메인 관리
    * 도메인 정보는 서브 도메인에 대한 쿠키의 접근을 관리하는데도 사용됨
    * 이를 통해 대규모 웹사이트에서 효율적으로 사용자 인증 및 선호도 설정을 관리할 수 있음
  * 규정 준수
    * 법적 요구사항을 준수하는데 필요함

### 쿠키는 어디에 저장될까? 디스크 or 메모리?
> 쿠키는 사용자의 컴퓨터에 저장되며,저장 위치는 쿠키의 유형에 따라 다르다
* **세션쿠키(Session Cookie)**
  * 세션 쿠키는 사용자의 세션이 활성 상태인 동안만 존재함
  * 이 쿠키는 주로 `메모리`에 저장되며 사용자가 브라우저를 닫으면 세션 쿠키는 삭제됨
* **영속쿠키(Persistent Cookie)**
  * 영속 쿠키는 설정된 만료 날짜가 지날 때 까지, 또는 사용자가 수동으로 삭제될 때 까지 컴퓨터의 하드 드라이브에 저장됨
  * 이 쿠키는 `디스크`에 저장되며, 브라우저를 닫아도 남아있음
  * 영속 쿠키는 사용자의 선호도, 로그인 정보등을 장기간 기억하는데 사용됨
  * 예) 로그인 상태 유지
* *이 둘의 차이는 지속성과 저장 위치에 있다*

### 쿠키는 우리가 서버에 요청을 보낼떄 자동으로 보내는데 왜그럴까?
> 웹의 상태 비저장(Stateless)특성과 관련이 있다. 서버는 각각의 HTTP요청을 독립적인 것으로 처리하는데    
> 이런 상태에서 사용자의 상태 정보나 세션을 유지하기 위해 쿠키가 사용됨
* 사용자 식별과 세션 관리
  * 서버는 쿠키를 사용하여 사용자를 식별하고, 사용자의 로그인 상태, 세션 정보, 이전 방문 정보 등을 유지할 수 있음
  * 예를 들어 사용자가 다른 페이지로 이동할 때 마다 해당 사용자를 인증하고 적절한 정보를 제공하는데 사용됨
* 상태 유지의 필요성
  * 상태 비저장 프로토콜인 HTTP에서 상태정보를 유지하기 위해 이 정보를 저장하는 쿠키를 요청마다 서버에 보내는 것이 필요
* 자동화 된 프로세스
  * 쿠키를 자동으로 전송하도록 함으로써, 사용자는 매번 인증 정보를 입력하거나 선호도를 설정할 필요 없이 원활한 웹 브라우징 경험을 할 수 있음
* 보안 및 개인화
  * 보안을 유지하고, 사용자 맞춤형 경험을 제공하기 위해, 서버는 쿠키를 통해 이전 활동 및 설정을 기억함

### 세션이 뭘까? 왜쓸까? 세션은 FE에서 어디에 저장될까?
> 세션은 특정 사용자가 웹 브라우저를 통해 웹 서버에 연결된 상태를 유지하는 기술이며,     
> 사용자가 웹 사이트에 로그인하거나 특정 웹사이트를 탐색하는 동안 사용자의 상태를 추적하고 관리하는데 용이함
* **세션의 사용목적**
  * 사용자 인증
    * 사용자가 로그인할 때 세션을 사용하여 사용자의 로그인 상태를 유지함
    * 사용자가 사이트의 다른 페이지로 이동할 때 마다 로그인 정보를 다시 입력할 필요가 없음
  * 상태 정보 유지
    * HTTP가 Stateless프로토콜 특성을 가지기 때문에 사용자가 사이트에 방문하는 동안 상태 정보를 유지하는데 사용함
  * 보안
    * 사용자의 중요한 정보를 서버측에서 관리하기 때문에 보안성이 높은 데이터를 클라이언트 측에서 직접 다루는 것 보다 안전함
* **저장위치**
  * 서버 측 저장
    * 서버의 메모리나 데이터베이스에 저장됨
    * 서버는 사용자마다 고유한 세션 ID를 생성하여 클라이언트에 전달하고 이를 통해 사용자의 세션 데이터에 접근
  * 클라이언트 측 저장
    * 세션 ID는 클라이언트 측에서 주로 쿠키에 저장됨
    * 브라우저는 매 요청마다 이 세션 ID를 포함한 쿠키를 서버에 전송하여 사용자를 식별함
  * 프론트엔드
    * 직접적으로 세션 데이터를 저장하지는 않으나, 세션 ID를 쿠키나 로컬 스토리지에 저장하여 서버와의 통신에 사용함
    * 실제 세션 데이터는 보안상의 이유로 서버측에 저장함
* **Session ID**
  * 사용자가 웹 사이트와의 세션을 시작할 때 생성 됨
  * 서버측에서 생성하며, 난수 생성기를 사용하여 생성됨

### JWT(JSON WEB TOKEN)방식
> RESTful API와 마이크로 서비스 아키텍처를 사용하는 애플리케이션에서 주로 선호함
* 상태 비저장
  * JWT는 상태 비저장 인증 방식으로 서버측에서 사용자의 인증 상태를 유지할 필요가 없어 서버의 세션관리 부담을 줄이며,     
    서버간의 부하 분산 및 확장성 측면에서 유리함
* 분산 시스템에 적합
  * 마이크로 서비스 및 분산 시스템 아키텍처에서 JWT는 서비스 간의 인증을 용이하게 함
  * 하나의 JWT가 여러 서비스에서 인증 수단으로 사용될 수 있으며,각 서비스는 독립적으로 토큰의 유효성을 검증할 수 있음
* 모바일 및 크로스-플랫폼 인증 지원
  * 다양한 클라이언트에서 인증 정보를 쉽게 교환할 수 있게 해줌
  * 모바일 앱과 웹 애플리케이션 간의 통합을 간편하게 함
* 보안성
  * JWT는 디지털 서명되어 있어 데이터의 무결성을 보장함
  * 토큰이 탈취되거나 변조되는 것을 방지함
* 확장성과 효율성
  * JWT는 클라이언트 측에 저장되며, 서버는 요청마다 토큰의 유효성만을 검증하면 되므로, 서버의 리소스 사용을 줄이고 확장성을 높일 수 있다.
* 개발의 용이성
  * JWT는 자체적으로 필요한 모든 정보를 담고있어, 별도의 데이터베이스 조회 없이 인증과 권한 부여가 가능함
* **단점**
  * 한번 발급된 JWT는 만료 시간이 도래하기 전까지는 무효화 시킬 수 없음
  * 토큰이 탈취될 경우 보안상의 위험이 있음
  * 따라서 JWT를 사용할 떄에는 보안 조치와 함께 적절한 만료 시간 설정, 토큰 리프레시 전략등을 고려하는 것이 중요

### 서버에서 발급된 Session ID가 브라우저에 저장되는 과정
* 세션 ID 생성(서버) -> 세션 ID 서버 응답에 포함(Set-Cookie필드) -> 브라우저에 쿠키 저장 -> 브라우저에서 서버로의 후속 요청 -> 세션 유지 및 관리

### JDBC 드라이버를 로딩할 때 `Class.forName` 이라는 메소드를 사용하는데 이 메소드는 어떤 역할을 하는가?
* 동적으로 클래스를 로드하는데 사용되는 메서드이다.
* 전달된 문자열 이름을 가진 클래스를 동적으로 로딩하여 런타임시에 클래스를 메모리에 로드함
* (드라이버 초기화)JDBC드라이버가 자동으로 DriverManager에 등록되고, 이후 데이터 베이스 연결 시 이를 통해 연결할 수 있게 됨
* Name을 통해 어떤 객체를 생성할지를 결정하는 메서드로, **팩터리 메서드 패턴**을 이용하여 만들어진 메서드이다

### 데이터베이스 호출을 끝내면 `ResultSet` , `Statement` , `Connection` 순서대로 close를 호출해야 하는 이유
* 의존성 관계
  * ResultSet은 Statement 객체에 의존하고, Statement는 Connection 객체에 의존함.
  * 즉, ResultSet은 Statement를 통해 생성되고, Statement는 Connection을 통해 생성됨
* 안전한 리소스 해제
  * ResultSet을 닫으면 관련된 데이터 처리가 완료되었음을 나타내며, 이후 Statement를 닫아 해당 SQL 명령어에 대한 처리를 종료
  * 마지막으로 Connection을 닫아 데이터베이스와의 연결을 해제. 
* 리소스 누수 방지
  * 만약 Connection을 먼저 닫으면, 아직 열려 있는 Statement와 ResultSet이 정상적으로 작동하지 않을 수 있다.
* 일관된 리소스 관리
  * 모든 리소스를 일관되고 예측 가능한 방식으로 관리함으로써, 리소스 관리의 복잡성을 줄이고 버그 발생 가능성을 낮출 수 있다.

### SQL Injection 이란?
> 보안 취약점 중 하나로, 악의적인 사용자가 웹 애플리케이션을 통해 데이터베이스 시스템에 비정상적인 SQL 명령을 주입하여 실행시키는 공격

### 커넥션풀과 스레드풀의 개수 적절하게 결정하는 법
* 서버의 하드웨어 자원
  * 사용 가능한 CPU 코어 수, 메모리 크기, 네트워크 대역폭 등이 스레드 풀과 커넥션 풀의 크기를 결정하는 데 중요한 요소.
  * 일반적으로 스레드 풀의 크기는 CPU 코어 수에 따라 조정되며, 너무 많은 스레드가 CPU를 과도하게 사용하면 컨텍스트 스위칭으로 인해 성능이 저하될 수 있다.
* 애플리케이션의 특성
  * 애플리케이션의 특성(예: CPU 집약적, IO 집약적)에 따라 스레드 풀의 최적 크기가 달라질 수 있다.
  * IO 집약적인 애플리케이션의 경우, 스레드가 IO 작업을 기다리는 동안 다른 스레드가 작업을 수행할 수 있으므로 더 많은 스레드가 유용할 수 있습니다.
* 트래픽 패턴
  * 애플리케이션의 트래픽 패턴(예: 피크 시간대의 요청 수, 요청의 복잡도)을 분석하여 스레드 풀과 커넥션 풀의 크기를 조정.
  * 트래픽이 많은 시간에는 더 많은 스레드와 커넥션을 필요로 할 수 있습니다
* 커넥션 풀과 스레드 풀의 비율
  * 일반적으로, 각 스레드는 하나의 데이터베이스 커넥션을 사용.
  * 스레드 풀의 크기가 커넥션 풀의 크기를 초과하지 않도록 하는 것이 좋다.
  * 실제 최적 비율은 애플리케이션의 특성과 요구 사항에 따라 달라질 수 있다.
    
### MVC 패턴이란?
> 세가지 주요 구성 요소로 분리하여 관리한다. 모델/뷰/컨트롤러로 나누며 이 패턴의 목적은     
> 애플리케이션 데이터처리, 사용자 인터페이스, 입력처리 간의 결합도를 낮추어 애플리케이션의 유연성, 확장성 및 유지보수성을 향상시키는 데에 있음
* MVC 패턴의 역할 분리 이점
* 역할이 분리되어 있어 애플리케이션의 유지보수가 용이해짐
* 모델과 비지니스 로직이 분리되어있어, 다양한 뷰를 사용하여 동일한 데이터 모델을 다른방식으로 표현 가능하며, 이는 재사용성과 확장성을 높임
* 컴포넌트들이 분리되어있어, 각가의 컴포넌트를 독립적으로 테스트하기 쉬움
* 애플리케이션의 결합도를 낮춰 한 부분의 변경이 다른 부분에 미치는 영향을 최소화함

### 커맨드 패턴이란?
> 행동 디자인 패턴의 한 종류로, 요청을 객체의 형태로 캡슐화하여 사용자가 요청을 보내는 쪽과 요청을 처리하는 쪽을 분리하는 것이 목적    
> 이 패턴은 명령을 실행하는 호출자와 실제 작업을 수행하는 수신자 사이의 결합도를 줄이며, 명령을 클래스로 표현하여 확장성과 유연성을 증가시킴
* 장점
  * 분리와 확장성
    * 명령의 발행과 실행을 분리함으로써 시스템의 다른 부분에 영향을 미치지 않고 새로운 명령을 쉽게 추가할 수 있음.
  * 재사용과 조합
    * 이미 정의된 커맨드를 재사용하거나 조합하여 복잡한 작업을 수행할 수 있음.
  * 취소 및 로깅 기능
    * 커맨드 패턴은 명령의 기록, 취소 또는 재실행과 같은 추가적인 기능을 구현하기 쉽게 해줌.
